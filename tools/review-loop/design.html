How I keep AI coding agents honest

Never trust the agent's own confidence — the gates are the signal. An agent will tell you it's done, confidently, while an edge case is quietly broken. So every change runs a gauntlet of independent reviews before it can land. Here is that gauntlet — and how you dial its strictness to what a project can afford to get wrong.

The five checks, explained

Each is independently switched on or off per project. The tags show when it runs and how hard it is to get around.

Second-agent review

before commitsoft

A fresh reviewer signs off before anything is committed.

A different agent than the one that wrote the code reviews it — no ego in the diff, so it actually finds things. The approval is locked to the exact code reviewed: change one line and it has to be reviewed again.

Different-vendor review

before commitsoft

A second opinion from another company's AI.

The same change is reviewed by a model from a different vendor. When two independent AIs flag the same bug, it's almost certainly real — that's the payoff of not letting one model grade its own homework.

Final pre-merge review

before mergesoft

The finished result is reviewed once more.

Ten commits that each looked fine can still add up to a problem, and code can change after the last review. So right before it merges, the whole thing is checked again — and re-blocked if anything moved since.

Un-skippable block

at commit, in gitHARD

Makes the review impossible to route around.

Instead of asking the agent to run the review, the block lives inside git itself — so it fires no matter which shortcut is taken, including the ones agents used to slip through. Overriding it takes a deliberate, visible human action.

Server backstop

after mergeserver-side

A final independent check no local trick can dodge.

A real build and audit run on the server after merge. Even a change that got in through an unusual path still has to pass here before it reaches anyone.

Dial the strictness — three presets

Rather than wiring each project by hand, you pick a preset. Each adds to the one before, so you turn strictness up to match what the project can afford to get wrong.

LIGHT

Lower-stakes apps where a bug is recoverable. One honest independent review is enough — more gates would be friction without payoff.

STANDARD

Most production apps. A second AI and a final pre-merge pass catch what one reviewer on one commit would miss.

MAXIMUM

Public or accuracy-critical work where a mistake simply can't ship. The review becomes impossible to skip, with a server as the last line.

The two things you'll actually change per project

# one small config file per project
{
  "preset": "standard",        # light | standard | maximum
  "merge_base": "main",         # the branch you merge into
  "vendor": "codex",            # which AI does the review
  "vendor_fallback": "deepseek"
}
↳ Change the branch it merges into

Set merge_base to your branch name. It defaults to main; point it at develop (or anything) when a project integrates elsewhere. The pre-merge check reads this — nothing is hardcoded.

↳ Change the AI supplier

Set vendor to whichever model reviews the code. vendor_fallback steps in when the first is rate-limited, so the review never silently skips. Set it to none to turn fallback off.
